## 2. What's new in v0.4.0

### Tiered LLM routing

Brain v0.3.0 used the 80B Qwen-Next-Instruct for every LLM call - overkill for HyDE rewrites, wrong tool for reasoning tasks. The new `LLMRouter` maps `task_type` onto three model tiers:

| Task | Model | Why |
| --- | --- | --- |
| `fast` (HyDE rewrites, simple expansions) | `gemma-4-26B-A4B-it-OptiQ-4bit` | ~16 GB, fast prompt processing |
| `summarize`, `chat` | `Qwen3-Next-80B-A3B-Instruct-4bit` | quality, cite-or-admit fidelity |
| `reasoning`, `decompose`, `judge` | `Qwen3-Next-80B-A3B-Thinking-4bit` | chain-of-thought decomposition |

Per-model lazy load via dict-keyed cache. Disabling the router (`LLM_ROUTING_ENABLED=False`) collapses every task back to `LLM_MODEL` for v0.3.0-compatible single-model deploys.

### Multi-query agentic retrieval

The new `multi_query_search` MCP tool handles compound / multi-hop questions where dense embedding alone collapses meaning. Pipeline:

1. **Decompose** via Thinking tier → JSON list of 3-5 focused sub-queries.
2. **Hybrid-search** each sub-query (dense + BM25 + RRF, the SDE-30 path).
3. **Fuse** the ranked lists via outer-RRF (k=60) on chunk id.
4. **Render** the decomposition + merged hits in Markdown for audit.

Robust JSON extraction: a chatty LLM preamble doesn't break us, malformed output falls back to bare-query single-leg search.

### Embedding upgrade

Dense embedder swapped to the 8B-4bit-DWQ Qwen3 model - top of MTEB at 70.58. The 4096-dim native output is Matryoshka-trained, so the leading 1024 floats retain ~95% of the semantic signal. We take the leading slice and keep the existing Qdrant schema, which means the upgrade is purely a re-embed - no collection structure change. Re-embed of 140 files / 2737 chunks took ~10 minutes on M5 Pro.

## 5. Lessons learned

### Technical

- **MRL truncation lets us upgrade embedders without schema migrations.** The 4096→1024 leading-slice approach kept the Qdrant collection unchanged. Re-embed alone, no collection shape change. That's the kind of compatibility the embedding model designers explicitly built for, and we benefit by leaning into it.
- **Tiered routing is mostly a dict.** The whole router is ~90 lines of Python; the value is in the *mapping*, not in any clever scheduling. Resist the urge to add heuristics until something concrete asks for them.
- **JSON extraction needs to be defensive.** Local LLMs respect "reply with JSON only" maybe 80% of the time. Always parse with a fallback and never let parse failure break the user-visible path. The tool surface degrades to bare-query gracefully.
- **Per-model lazy load avoids memory pressure in practice.** With three tiers configured, daemon boot still loads zero LLMs. First call to each tier pays its cold start. Concurrent requests spanning two tiers (gemma-4 + Qwen-Next-Instruct ≈ 61 GB) stay under the 64 GB box; a third tier at the same time would evict, which the OS handles cleanly.

### Process

- **Honest scope reduction up front beats heroic completionism.** The Tier 4 prompt asked for 8 issues; I committed to 3-4 with explicit rationale and shipped them clean. The four deferred issues are documented in Linear with "reopen when X" criteria - same engineering value, shorter risk window.
- **Snapshot + git tag before every destructive op.** Once again the rollback path was a single command. The pattern is now muscle memory.
- **MCP tool descriptions earn their keep.** The "Use when..." sentence on `multi_query_search` is what tells an LLM client to stop firing it for trivial single-topic queries. Without that disambiguation, every tool collapses to the cheapest one.

## 6. Future backlog

### High-leverage if reopened

- **SDE-67 GraphRAG-lite** - entity/relation extraction over the corpus would give us a `find_related(entity)` tool that the existing keyword filter can't match. Cost: 90-180 min batch + ~50 lines of payload wiring. Reopen with eval-shaped time and a checkpoint/resume scaffold.
- **SDE-66 Reranker upgrade** - only worthwhile if a meaningful fraction of queries currently bottleneck on jinaai's quality. Track query-level dissatisfaction first; reopen when there's evidence.

### YAGNI for now

- **SDE-68 PDF ingestion** - DeepSeek-OCR-2-6bit is downloaded and ready, but the brand corpus is ~all Markdown. Reopen when a Figma export, vendor brief, or sprint-report PDF lands and motivates the integration.
- **SDE-69 Image ingestion** - same calculus. Series #02+ case studies might bring the corpus over the threshold.

### Engineering hygiene

- **Eval suite (SDE-34 reactivation)** - now that the LLM router is in place, a custom eval scaffold using the Thinking tier as judge is a clean fit. The router gives us isolated test access.
- **Memory budgeting telemetry** - the router currently doesn't track which models are actually resident in RAM. Adding a `loaded_models()` count + an MCP `health` field would help an operator pick when to bounce the daemon.
