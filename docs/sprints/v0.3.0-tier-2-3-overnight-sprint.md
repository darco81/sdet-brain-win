## 4. Lessons learned

### Technical

- **Named-vector migration is cheap with snapshots.** The actual destructive work is one `DELETE /collections/...` + a re-ingest. Snapshots make it ~1.5 GB of disk for the rollback option, which is the right tradeoff at this scale.
- **RRF is the no-tuning fusion.** Cosine and BM25 scores live on totally different scales; if you weighted-sum them you'd be hand-tuning the weight forever. RRF only ranks, so it's robust by construction.
- **HyDE works for terse Polish queries.** A bare two-word Polish query embeds further from the corpus than the LLM-written hypothetical that knows the corpus's voice.
- **mypy + invariant generics in qdrant-client** are the recurring papercut. The right move is `cast(Any, ...)` at the call site, not type gymnastics - the runtime is correct, the SDK's union types are just exhaustive.
- **Lazy-load LLMs survive autonomous runs.** The 80B model never gets invoked during the sprint; tests use a fake `ILLM` and the daemon boot stays cheap. The user pays the cold start exactly when they ask the first question.
- **Stateless chat = no garbage to collect.** Keeping conversation history client-side is the simplest thing that works and makes the server trivially horizontally scalable later.

### Process

- **Pre-resolved decisions unlock autonomy.** The continuation prompt resolved the three ambiguous Block 2/3 decisions explicitly (in-place wipe, MLX-only, local LLM judge) - that's what made running these issues without supervision safe.
- **Tar + Qdrant snapshot before every destructive op.** Twice now (SDE-28 corpus migration, SDE-30 collection wipe). Both rolled forward fine; both would have been recoverable in <15 min from the snapshots if not.
- **Hard-stop conditions in the prompt prevent rabbit-holing.** "If RAGAS doesn't work AND custom scaffold takes >30 min, defer" was the exact rule that protected the closing window.

### What I'd retro

- The migration tool's "merge" mode for SDE-28 grew to 11 lines of status remap (`in-progress` → `draft`, `completed` → `published`, etc.). That's user data normalisation that the schema *doesn't* enforce - easy to drift if more values appear. Worth a follow-up: extend the schema's `Status` literal to cover the corpus's actual statuses, drop the remap.
- Tests for the LLM-backed tools lean entirely on monkeypatched fakes. Unsurprising (real weights won't fit in CI), but means the production prompts are unverified except by manual eyeball. SDE-34 (RAGAS) would close this loop.
- Health check on the LLM provider returns False until a real call lands. That's correct semantically but confusing for a `/health` consumer who'd expect "ready or not". A `prewarm=true` flag on `init_collections` could opt-in load the LLM at boot for environments that prefer the latency.

## 5. Future backlog

### Tier 3 follow-ups (scoped)

- **SDE-34 (deferred)** - RAGAS or custom eval scaffold using local Qwen as judge. ~2-3h once started.
- **SDE-35** - VPS deploy. Requires SSH + production env. Sprint 13-16.05 per the master plan.
- **SDE-45** - `keyboard-trap-runtime` detector exemption rule (filed during Series #01 prep, not brain).

### Engineering hygiene

- **Status enum expansion** - fold the migration tool's remap into `BrandFrontmatter.Status` so the schema covers the corpus's actual values and the migration becomes pure validation.
- **Streaming reranker** - when SDE-31 + SDE-30 are both active, the reranker reorders dense+sparse fused candidates. Minor optimisation: reranker's prefetch limit could match `RERANK_TOP_K_RETRIEVE` instead of being independent.
- **HTML test client polish** - the chat-test.html is functional but spartan. Could grow into the `portfolio.sdet.it/brain` demo (T3-03 territory).

### Bigger ideas

- **Function-calling for the local LLM.** Right now the LLM is a generation endpoint; tools live one layer up. A local-tool-loop would let `/chat` itself route through `query_rewrite` or `summarize_results` without the user knowing.
- **Per-source-type embedders.** Articles in EN benefit from one embedder; Polish drafts from another. We pin to one MLX dense model today; a second named vector (`dense_en`, `dense_pl`) is straightforward but adds storage cost.
