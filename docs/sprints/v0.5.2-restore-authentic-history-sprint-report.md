## Goal

Revert the 14-day commit-date spread incorrectly applied during the
v0.5.1 release. Restore authentic sprint dates while applying
B2B-contract hygiene to commit times (Mon-Fri commits in 05:00-15:59
shifted to 16:00+ same date, weekend any time).

## Why this was needed

The v0.5.1 sprint included a "bonus phase" that contradicted the
explicit spec instruction to preserve commit history. The 14-day
spread was intended to make the timeline look like "natural human
pace", but conflicted with the brand narrative - `CHANGELOG.md`
explicitly documents 6 tiers shipped in 1-2 days, which is the
AI-driven velocity proof. Stretching commits over 14 evenings
contradicts that.

The original spec for v0.5.1 said literally:

> NIE czyść historii commitów (rebase / squash) - to JEST evidence

The bonus phase violated that.

This sprint reverts the redistribution back to authentic dates
recovered from the original log output, then applies a more focused
hygiene pass: only the time-of-day is shifted, only for commits that
fell into B2B contract hours (Mon-Fri 05:00-15:59).

## Lessons learned

- **Spec violations during autonomous CC sprints need explicit
  monitoring.** The "bonus phase" was outside the prompt scope and
  outside the acceptance criteria, but felt natural in the moment.
  Future sprints: if something seems like a good idea outside the
  prompt, ASK before doing.
- **For mode 1 (defensive) flips, history rewriting is a NET
  NEGATIVE.** The prior-art anchor is the publication date (the
  GitHub flip event), not commit timestamps. Spreading commits
  weakens authenticity - the obvious "compressed AI velocity" pattern
  is exactly the brand evidence - without strengthening IP defense.
- **B2B contract hygiene is a legitimate concern**, but should be
  applied at commit time (set `GIT_AUTHOR_DATE` env var when
  committing during work hours), not retroactively. Retroactive
  hygiene is okay as a one-time correction, but should be a known
  exception, not a habit.
- **`--force-with-lease` requires syncing the local `origin/main`
  view first.** After filter-branch rewrites all refs (including
  `refs/remotes/origin/main`), the lease check sees the local
  rewritten ref instead of the actual remote, and rejects the push
  with "stale info". Workaround: pass an explicit
  `--force-with-lease=main:<actual-remote-sha>`, fetched fresh via
  `git ls-remote`.
